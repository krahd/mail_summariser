What the current source actually does

Source-aligned feature set

Split message review

The current main workspace keeps the message list on the left and opens the selected mail on the right, so you can read sender and recipient details plus the body without leaving the summary context.

Two mailbox paths

Use the in-memory dummy mailbox to validate the workflow quickly, then switch dummyMode off and point the same UI at a real IMAP inbox and SMTP server. Dummy-mode jobs, logs, undo entries, mailbox state, and outbox are reset whenever the backend restarts or the mode changes.

Provider fallback built in

Summaries can run through Ollama, OpenAI, or Anthropic. If the provider is unavailable, returns invalid output, or sends placeholder text, the backend falls back to a deterministic digest.

Traceable actions

Live-mail summary jobs, follow-up actions, and undoable operations are persisted so the log view can show what happened and whether a row is still reversible, while dummy-mode activity stays isolated from the database.

Local fake mailbox

When MAIL_SUMMARISER_ENABLE_DEV_TOOLS=true, both clients can start a localhost-only fake IMAP/SMTP server, copy disposable credentials into Settings, and exercise the live-mail flow without leaving the app surface.

Model management

The browser and macOS settings surfaces can check Ollama availability, offer install or start actions on startup, list installed models, browse a downloadable catalogue, and request local pulls without leaving the app surface.

Workflow

From dummy mailbox to real account

The current codebase is structured so you can validate the whole experience locally first, then reuse the same actions and logging flow against a real mailbox.

  1. 01

    Start the backend

    Run ./start_backend.sh, then open /web in a browser or launch the macOS app. If Ollama is your saved provider, mail_summariser now checks whether it needs to be installed or started.

  2. 02

    Validate in dummy mode

    Search the built-in messages, create a summary, review the selected mail in the side-by-side detail pane, and try Mark Read, Tag Summarised, and Email Summary. These dummy-mode artifacts stay in an in-memory sandbox and never touch the SQLite database.

  3. 03

    Configure providers and mail settings

    Use the browser Settings tab or the macOS Settings window to point at Ollama, OpenAI, or Anthropic, save IMAP/SMTP details, reset the local database if needed, and choose whether Ollama should start on app startup or stop on exit.

  4. 04

    Switch to live mail

    Turn dummy mode off, run Test Connection, or start the dev-only fake IMAP/SMTP server and apply its suggested settings, then repeat the same summary and action flow against that disposable mailbox.

Client surfaces

Browser and macOS

The browser client exposes the fuller operator surface today. The SwiftUI app is intentionally thinner and relies on the same backend API.

Browser client

  • Main: search criteria, summary generation, split message review with selected-mail detail, and action buttons.
  • Log: action history with final versus undoable state.
  • Settings: dummy mode toggle, connection test, provider keys, Ollama startup and exit controls, model management, database reset, fake mail server tools, backend URL, backend API key, and backend shutdown.
  • Help: in-product user manual and project links.

macOS app

  • Main tab: summary workflow, split message review with selected-mail detail, and action buttons.
  • Log tab: log table with per-row undo.
  • Settings window: separate macOS Settings scene for mail, provider configuration, Ollama lifecycle controls, database reset, fake mail server tools, and backend shutdown.
  • Backend dependency: the desktop app is a client for the FastAPI service, not a bundled all-in-one runtime.
